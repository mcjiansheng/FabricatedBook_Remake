# AI Agent 项目规约

本文件是本仓库面向 AI Agent 的项目级工作规约。任何 Agent 在修改代码、测试、数据或文档前，都应阅读并遵守本文件。

## 复杂功能开发协议

当任务涉及新增复杂功能、修改核心游戏规则、跨多个模块改动、存档/数据格式变化，或影响玩家可见流程时，Agent 必须遵守以下协议。

### 1. 先明确需求

- 在实现前先理解用户期望的具体行为。
- 如果需求、边界条件、优先级、游戏机制或验收标准存在不确定，必须暂停并向用户询问。
- 不得在玩法机制、奖励规则、卡牌效果、存档兼容、UI 流程等关键问题上静默假设。
- 如果实现依赖重要假设，应在执行前或执行过程中明确说明。

### 2. 明确规划流程

- 在进行实质性编辑前，将任务拆成清晰的执行步骤。
- 计划必须基于当前代码结构，而不是脱离项目重新设计。
- 如果调查代码后发现原计划不合适，应更新计划再继续。
- 默认执行顺序为：阅读现有实现、确认复用点、修改核心逻辑、同步 UI/数据/文档、补充或调整测试、运行验证、总结结果。

### 3. 复用现有架构

- 新增功能前必须阅读并理解相关已有库、工具、方法、类和数据流。
- 优先使用项目已有模式，不创建重复工具、重复管理器或重复规则系统。
- 不得绕过已有的工厂、管理器、计算器、实体模型、UI helper 或数据加载器，除非明确说明原因。
- 新概念必须接入已有职责边界：战斗规则属于 core engine/entity，数值预览属于 preview calculator，界面表现属于 view actor/screen，数据读取属于 factory/loader。

### 4. 保持可扩展和面向对象

- 按领域对象和层级划分职责，避免把核心规则塞进 UI 或无关 manager。
- 新状态和行为应放在真正拥有该概念的对象上。
- 避免只为当前一张牌、一个敌人、一个界面写硬编码特例。
- 新方法和字段应能支持未来更多卡牌、藏品、Buff、敌人、事件和存档数据。
- 修改影响到运行逻辑、数值预览、测试、存档/数据格式、UI 文案或文档时，应同步覆盖这些表面。

## 验证要求

- 根据改动范围运行最小但有效的测试；涉及共享系统时应扩大测试范围。
- 前端或 UI 改动应尽量使用项目文档中的前端调试方法进行检查。
- 如果无法验证，必须明确说明未运行什么验证以及原因。

## Git 与工作区安全

- 不得回退或覆盖用户/其他协作者的未提交修改，除非用户明确要求。
- Commit 应聚焦当前任务，不混入无关修改。
- 如果用户要求实时 commit，应在完成一个经过验证的逻辑单元后及时提交。
