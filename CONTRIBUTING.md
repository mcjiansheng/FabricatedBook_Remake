# GitHub 在线合作代码管理协议

## 1. 目的

为保证项目在 GitHub 上的协作效率、代码质量、版本稳定性和责任可追溯性，所有参与本项目的编辑者、维护者、开发者以及自动化 Agent，均应遵守本协议。

本协议适用于本仓库内的全部代码、文档、配置、Issue、Pull Request、评论、自动化提交和发布流程。

## 2. 适用对象

本协议适用于：

- 项目 Owner
- Maintainer
- Collaborator
- Contributor
- 受邀编辑者
- AI Agent、代码生成 Agent、自动修复 Agent、CI Agent 等自动化参与者

任何人或 Agent 一旦对仓库进行修改、提交、评论、审查或合并，即视为同意遵守本协议。

## 3. 基本协作原则

所有协作者应遵循以下原则：

- 不直接破坏主分支稳定性。
- 不提交未经验证的代码。
- 不覆盖他人正在进行的工作。
- 不隐藏、伪造或模糊变更来源。
- 不绕过必要的 Review、测试和 CI 流程。
- 不将密钥、账号、Token、隐私数据或敏感配置提交到仓库。
- 所有重要变更必须可追踪、可解释、可回滚。

## 4. 分支管理规范

### 4.1 受保护分支

以下分支应视为受保护分支：

- `main`
- `master`
- `release/*`
- `production`
- 其他由维护者指定的稳定分支

受保护分支禁止直接 Push，除非项目 Owner 明确授权。

### 4.2 工作分支命名

所有功能、修复和实验性修改应在独立分支完成。

推荐命名格式：

```text
feature/功能名称
fix/问题名称
docs/文档名称
refactor/重构名称
test/测试名称
chore/维护任务
agent/agent名称-任务名称
```

示例：

```text
feature/user-auth
fix/login-error
docs/api-guide
agent/codex-fix-ci
```

### 4.3 分支使用要求

- 一个分支应对应一个明确任务。
- 不应在同一分支混合多个无关变更。
- 长期分支应定期同步目标分支。
- 废弃分支应在确认无用后删除。

## 5. Commit 提交规范

### 5.1 Commit 信息格式

推荐使用以下格式：

```text
类型: 简短描述
```

常用类型：

```text
feat: 新功能
fix: 修复问题
docs: 文档修改
style: 格式调整，不影响逻辑
refactor: 重构
test: 测试相关
chore: 构建、依赖、配置等维护工作
perf: 性能优化
ci: CI/CD 配置修改
revert: 回滚提交
```

示例：

```text
feat: add user registration page
fix: correct token refresh logic
docs: update deployment guide
```

### 5.2 Commit 要求

每个 Commit 应满足：

- 变更范围清晰。
- 不包含无关文件。
- 不包含临时调试代码。
- 不包含敏感信息。
- 不提交本地环境生成文件，除非项目明确需要。
- 自动化 Agent 提交时，应在 Commit 或 PR 描述中说明其身份和任务来源。

## 6. Pull Request 规范

### 6.1 PR 创建要求

所有进入受保护分支的修改必须通过 Pull Request。

PR 应包含：

- 变更目的
- 主要修改内容
- 影响范围
- 测试方式
- 相关 Issue 或任务链接
- 如有 UI 修改，应提供截图或说明

推荐 PR 模板：

```md
## 变更说明

简要说明本次修改的目的和内容。

## 修改类型

- [ ] 新功能
- [ ] Bug 修复
- [ ] 文档修改
- [ ] 重构
- [ ] 测试
- [ ] 配置/构建
- [ ] 其他

## 测试情况

说明已执行的测试：

- [ ] 本地测试通过
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] CI 通过
- [ ] 不适用，原因：

## 影响范围

说明本次修改可能影响的模块、接口、页面或流程。

## 关联任务

Closes #
Related to #
```

### 6.2 PR 审查要求

PR 合并前应满足：

- 至少经过一名维护者或指定 Reviewer 审查。
- CI 检查通过。
- 无未解决的阻塞性评论。
- 无明显冲突。
- 涉及核心逻辑、安全、数据结构、权限或发布流程的修改，应进行更严格审查。

### 6.3 PR 合并要求

合并方式由维护者决定，可使用：

- Squash merge
- Merge commit
- Rebase merge

合并后应根据需要删除源分支。

## 7. Issue 管理规范

Issue 用于记录：

- Bug
- Feature request
- 任务拆分
- 技术讨论
- 重构计划
- 文档改进
- Agent 执行任务

Issue 应尽量包含：

- 背景
- 期望结果
- 当前问题
- 复现步骤
- 截图、日志或相关链接
- 优先级或标签

禁止使用 Issue 进行无关争论、攻击性表达或与项目无关的内容。

## 8. Code Review 规范

Reviewer 应重点关注：

- 代码是否解决了目标问题。
- 是否引入新 Bug。
- 是否影响已有功能。
- 是否存在安全风险。
- 是否需要测试。
- 是否符合项目结构和风格。
- 是否存在重复代码或不必要复杂度。
- 是否有文档同步需求。

Review 评论应具体、可执行、尊重他人。

被 Review 者应：

- 及时回应评论。
- 对合理建议进行修改。
- 如不同意建议，应说明原因。
- 不应在未解决关键问题前强行请求合并。

## 9. 自动化 Agent 协作规范

所有参与本项目的 Agent 必须遵守以下规则。

根目录 `AGENTS.md` 是本仓库的项目级 AI Agent 工作规约。Agent 在执行新增复杂功能、核心规则变更、多模块改动或用户可见流程调整前，必须先阅读并遵循其中的“复杂功能开发协议”。

### 9.1 Agent 身份标识

Agent 在创建 Commit、PR、Issue 或评论时，应清楚标明：

- Agent 名称
- 执行任务
- 变更范围
- 是否经过测试
- 是否存在未完成事项

### 9.2 Agent 修改限制

Agent 不得擅自执行以下操作，除非得到明确授权：

- 删除大量文件
- 重写 Git 历史
- 强制 Push
- 修改权限配置
- 修改发布凭证
- 修改生产环境配置
- 合并 PR
- 发布版本
- 删除分支或 Tag
- 修改 License
- 修改安全策略

### 9.3 Agent 工作要求

Agent 应做到：

- 优先阅读现有代码和文档。
- 新增复杂功能时，先澄清需求、列出执行计划，再按计划实施。
- 遇到需求、边界条件或玩法规则不确定时暂停并询问用户。
- 理解并复用已有库、工具、方法和类，不重复设计已有能力。
- 保持可扩展性，按照清晰的面向对象职责边界实现。
- 遵循项目已有风格。
- 控制变更范围。
- 不进行无关重构。
- 不覆盖用户或其他协作者的未提交修改。
- 修改后尽可能运行相关测试。
- 在 PR 或总结中说明验证结果。
- 遇到不确定情况时提出说明，而不是假设性破坏修改。

### 9.4 Agent 生成内容要求

Agent 生成的代码、文档和配置应：

- 可读、可维护。
- 不包含虚假信息。
- 不引用不存在的接口、文件或依赖。
- 不引入未声明依赖。
- 不包含敏感信息。
- 不制造隐藏行为或后门。

## 10. 测试与质量要求

提交前应根据修改类型执行适当测试：

- 代码逻辑修改：运行相关单元测试。
- API 修改：验证接口兼容性。
- UI 修改：进行页面检查或截图验证。
- 构建配置修改：运行构建命令。
- 依赖修改：确认依赖可安装且无明显冲突。
- 安全相关修改：进行额外审查。

未经测试的修改应在 PR 中明确说明原因和风险。

## 11. 文档同步要求

当变更影响以下内容时，应同步更新文档：

- 安装方式
- 启动方式
- 配置项
- API
- 数据结构
- 用户流程
- 部署流程
- 权限模型
- 开发规范

文档应与实际代码保持一致。

## 12. 安全与敏感信息规范

禁止提交：

- 密码
- API Key
- Token
- 私钥
- 数据库连接凭证
- 用户隐私数据
- 生产环境配置
- 未脱敏日志

如发现敏感信息已提交，应立即：

1. 通知维护者。
2. 撤销或轮换相关密钥。
3. 清理 Git 历史或采取安全补救措施。
4. 记录事故处理过程。

## 13. 依赖管理规范

新增依赖前应确认：

- 是否确有必要。
- 是否有维护活跃度。
- License 是否兼容。
- 是否增加明显安全风险。
- 是否影响构建体积或性能。
- 是否已有项目内替代方案。

不得随意升级核心依赖。重大依赖升级应单独提交，并说明兼容性影响。

## 14. 发布管理规范

发布版本应由维护者执行。

发布前应确认：

- 目标分支稳定。
- CI 通过。
- 关键功能已验证。
- 版本号符合项目规范。
- Changelog 已更新。
- 必要文档已同步。
- 无未解决的阻塞 Issue。

Tag 命名推荐：

```text
v1.0.0
v1.2.3
```

## 15. 冲突处理

当出现代码冲突、职责冲突或实现方案分歧时，应优先通过 Issue、PR 评论或项目会议沟通解决。

处理原则：

- 以项目目标为准。
- 以代码质量和长期维护性为准。
- 尊重已有架构和历史背景。
- 重大分歧由 Maintainer 或 Owner 最终裁决。

## 16. 禁止行为

任何协作者或 Agent 不得：

- 未经授权修改受保护分支。
- 强制 Push 覆盖他人工作。
- 删除他人代码且不说明原因。
- 提交恶意代码、后门或隐藏逻辑。
- 提交敏感信息。
- 绕过 Review 和 CI。
- 伪造他人身份或贡献记录。
- 使用攻击性、不尊重或歧视性语言。
- 利用 Agent 自动生成大量低质量 PR 或 Issue。
- 在未经确认的情况下进行大规模格式化或重构。

## 17. 违规处理

违反本协议的行为，维护者可采取以下措施：

- 要求修改或回滚。
- 关闭 PR 或 Issue。
- 撤销写权限。
- 暂停 Agent 使用权限。
- 锁定讨论。
- 回滚合并。
- 在严重情况下移除协作者身份。

如违规造成安全、数据或生产事故，相关人员应配合调查和修复。

## 18. 协议维护

本协议由项目维护者负责解释和更新。

协议修改应通过 Pull Request 进行，并说明修改原因。重大规则调整应通知所有主要协作者。

## 19. 生效

本协议自合并至仓库之日起生效。

所有对本仓库进行编辑、提交、审查、评论、自动化修改或合并操作的人员及 Agent，均视为已阅读、理解并同意遵守本协议。
